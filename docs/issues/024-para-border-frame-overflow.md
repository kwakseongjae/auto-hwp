- 선행: 020 (done — --rows 진단도구 재사용). 병렬 가능: 025 (파일 disjoint)

## 사용자 QA 관찰 (benchmark1 자가진단표, 원본 PDF 대비)
(1) "위 사항을…동의합니다. 신청인 (서명)" 행이 상자여야 하는데 아님 (2) "[신청자 체크항목]"
헤더행과 마지막 "동의합니다" 문단이 상자여야 하는데 아님 (3) "중소벤처기업진흥공단 이사장
귀하"가 외곽 상자를 벗어남.

## 아키텍트 진단 (2026-07-03 실측 — 이 위에서 시작하라)
- **원인 A (박스 누락)**: `ParaShape.border_fill_id`(hwp-model style.rs:192)가 **파싱만 되고
  어디서도 소비되지 않는다** — lift는 cell의 border_fill_id만 사용(lift.rs:360-405), typeset/
  render에 문단 테두리 개념 자체가 없음. 상자 3종은 문단 테두리로 추정(텍스트는 나오고 박스만
  없음). ※단, 1행짜리 중첩표일 가능성도 완전히 배제 못 함 — **1단계에서 실체를 실측 확정**하라.
- **원인 B (귀하 이탈→소실)**: 원본은 귀하까지 **전부 외곽 상자 안**이다. 우리는 프레임(1×1 표)
  예약 높이 < 내부 콘텐츠일 때 초과 라인을 조용히 드롭한다 — **현재 native own-render 18페이지
  전체에서 "귀하" 문자열이 0건**(아키텍트 grep 실측; 사용자 화면의 '박스 밖 유출'은 구빌드).
  정답은 "프레임이 내부 콘텐츠만큼 성장"이며, 한컴도 그 내용을 포함하고 18쪽이므로 성장시켜도
  18이 유지되는 것이 정합이다(어긋나면 높이 계산의 다른 항이 틀린 것 — 020 도구로 실측).

## 목표
자가진단표가 원본과 동일하게: 상자 3종 렌더 + 귀하 포함 전부 외곽 상자 안. **텍스트 무소실**
(회귀 테스트로 고정). 게이트 v2 유지(benchmark 8==8·98.9%, benchmark1 **18==18**·99.2%).

## 파일 지도
- `crates/hwp-model/src/style.rs` — ParaShape.border_fill_id (이미 존재)
- `crates/hwp-rhwp/src/lift.rs` — 문단 테두리/배경 lift (cell_borders 재사용 지점) + 프레임 내부 구조
- `crates/hwp-typeset/src/lib.rs` + `place.rs` — 문단 박스 높이 기여(여백/패딩 실측 기반, **양쪽
  동일** — 020의 공유 헬퍼 패턴), 프레임 셀 높이=내부 콘텐츠 성장, 오버플로 드롭 제거
- `crates/hwp-render/src/lib.rs` — 문단 테두리/배경 rect 방출(셀 edge 경로 재사용)
- 진단: `layout-check --rows`(020 도구) + 필요 시 프레임 내부 블록 덤프 확장

## 구현 단계 (측정 우선 — 020과 같은 규율)
1. **구조 실측**: benchmark1 자가진단표 프레임 내부 블록 목록을 덤프(kind/텍스트 머리/
   border_fill_id/중첩표 여부)해 상자 3종의 실체(문단테두리 vs 1행표)와 귀하까지의 전체 콘텐츠
   시퀀스를 `docs/BENCHMARK1-ROW-AUDIT.md`에 추가 기록. 한컴 lineseg로 각 항목의 실제 높이/
   위치도 함께(020 도구 확장 허용 — tracked).
2. **문단 테두리 lift**: ParaShape.border_fill_id → Paragraph 레벨 테두리/배경으로 lift
   (cell_borders/cell_shade 함수 재사용). 모델에 필드 추가 필요 시 additive로.
3. **렌더**: 문단 박스 rect(4변 개별, 배경) 방출. **높이 기여는 실측으로**: 한컴이 문단테두리에
   주는 상하 간격을 lineseg 대조로 확인해 place/NaiveLayout 양쪽 동일 반영(임의 상수 금지).
4. **프레임 성장 + 무소실**: 프레임(래퍼 표) 셀 높이 = max(stored, 내부 콘텐츠 실측)로 성장,
   초과 라인 드롭 경로 제거. 분할이 필요한 높이면 기존 행 분할 메커니즘을 따르되 이 프레임은
   원본상 1페이지 수납이 정답.
5. **회귀 테스트**: (a) own-render benchmark1 전 페이지에 "귀하" 존재 assert(텍스트 무소실 —
   tracked 테스트), (b) 게이트 v2, (c) --rows 재실측으로 상자 3종 높이 델타 ±0 근접 확인.
6. golden 재고정(의도된 SVG 변화 — 상자 추가/프레임 성장 명시).

## 함정
- 문단 테두리 높이 기여를 넣으면 **다른 문서의 페이지가 밀릴 수 있다** — 게이트 두 문서로
  잡히지 않는 코퍼스 회귀는 오라클 스모크로 확인. 높이 기여가 0인 문서(테두리 없음)는 바이트
  불변이어야 한다(020/022의 골든 규율).
- 프레임 성장이 benchmark1을 19로 되돌리면 안 된다 — 한컴이 같은 내용으로 18을 내는 이상,
  19가 나오면 성장분 계산이 과대한 것(실측으로 어느 항인지 특정).
- rhwp 원본 무수정(lift는 우리 어댑터).
