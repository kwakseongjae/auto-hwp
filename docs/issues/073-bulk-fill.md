# 073 — 벌크 채움(HWP 메일머지): 원본 양식 + 명단 → 완성본 N부 zip

- 상태: open (타당성 실증 완료 — 설계/구현 대기) · 우선순위: 로드맵 축 후보 · 영역: 신규 레인(CLI 우선, 로컬)
- 발단(2026-07-23 사용자): 회사에서 100명분 정보(이름·계약기간·관계·서명)를 기관 제출용 한글 양식에
  채워야 했음 — "원본 hwp + 채우기 쉬운 구조의 데이터 → 완성본 벌크 zip"을 로컬에서.

## 타당성 실증 (2026-07-23 PoC — 실물 청창사 신청서 .hwp/.hwpx, @auto-hwp/engine Node)
1. **핵심 플로우는 지금 됩니다**: 템플릿 1회 인스펙션(docProfile+tableGrid로 라벨→값칸 fill-map 자동
   유도) → 인원별 open→SetTableCell→toHwpx → zip. **60ms/부**(100명 ≈ 6초). 산출물 재개봉 검증에서
   채운 값 3/3 실셀 확인.
2. **HWPX 템플릿 = 프로덕션급**: 무편집 왕복 **바이트 동일**, 편집 1건 후에도 쪽수 불변(17==17) —
   round-trip moat가 그대로 벌크 품질 보증이 된다. 온나라 HWPX 의무화(2026-05)로 기관 양식의 HWPX
   비중도 커지는 중.
3. **.hwp 템플릿 = 품질 축 잔여**: 채움은 되지만 산출물이 HWPX 변환본이라 실물 신청서에서 쪽수
   +1(18→19) 리플로 실측 — **편집 탓이 아니라 변환 축**(무편집 왕복만으로 재현). 완성본 충실도가
   목적이면 이 축(F2 계보)이 남은 격차의 전부다.
4. **실물 함정 실측**: ① 기관 양식 값칸은 빈칸이 아니라 예시 텍스트(김곤충 등)로 차 있음 — fill-map
   규칙은 "라벨 우측 첫 값칸 교체"(빈칸 전제 금지). ② .hwp→HWPX 저장 후 블록 인덱스가 재배열됨 —
   fill-map은 템플릿 기준 1회 유도·산출물 검증은 주소 재사용이 아니라 값 스캔으로. ③ 병합 커버 셀은
   편집 거부(정직) — active 셀만 대상.

## 갭 (정직)
- **바이너리 .hwp 출력 불가**(엔진 저장 = HWPX). 기관이 엄격히 .hwp만 받으면 미충족 — 단 한글이
  HWPX를 네이티브로 열고 정부는 HWPX로 이동 중. 생태계 조사 결과(아래): 무한컴 .hwp 출력이 가능한
  성숙 기존물은 hwplib(Java) 정도 — 필요해지면 kordoc `patchHwp`의 CFB 섹터 수술 접근(MIT)이 참고
  1순위.
- **누름틀(click-here) 채우기 op 없음** — 필드 파싱(FieldBegin)은 되나 쓰기 미지원. 누름틀 기반
  양식은 후속(대부분의 기관 양식은 표 셀 기반이라 v1은 셀+Replace로 충분).
- **서명 이미지**: InsertImage(블록 앵커)는 있으나 셀 안 도장/서명 스탬핑은 미지원 — v2 후보.
- .hwp 템플릿 충실도(위 3) · 실 한컴 오라클 미검증(기존 U12)은 이 유스케이스에서 체감 리스크가 가장 큼
  (제출물이니 기관 PC의 한글에서 열림새가 전부).

## 생태계 조사 (2026-07-23 웹 실측 — 상세 표는 조사 리포트)
- **"원본 .hwp 무훼손 + 무한컴(맥/리눅스) + 벌크 + MCP"를 한 번에 갖춘 기존물은 없다** — 공백지대.
- 반쪽씩 가진 것들: **hwplib**(Java, 유일한 성숙 무한컴 .hwp in→out + 누름틀 setFieldText — 단 전용
  메일머지 API 없음·복잡문서 저장 안정성 케이스별) · **hwpx-mcp-server**(`mail_merge` MCP 도구가
  유스케이스 직격 — 단 **HWPX 전용**, .hwp는 한컴으로 사전 변환) · **pyhwpx**(엑셀→put_field_text
  국내 표준 워크플로 — Windows+한컴 필수라 로컬 맥/리눅스 탈락) · **kordoc**(fill_form+CFB 바이너리
  패치 주장, 교집합에 가장 근접 — 신생·벌크는 외부 루프) · treesoop/hwp-mcp(쓰기 .hwpx만) ·
  hwpkit(주장은 직격, star 1 미검증).
- 시사점: 우리 v1(.hwpx 출력)은 hwpx-mcp-server의 mail_merge와 포지션이 겹치지만, **.hwp 입력을
  한컴 없이 받는 것 + 게이트/바이트 보존 검증 리포트 + 4표면(CLI·MCP·스킬·웹)**이 차별. 누름틀 fill
  op을 엔진에 넣으면(파싱은 이미 됨) pyhwpx의 워크플로를 무한컴·크로스플랫폼으로 흡수한다.

## 판단: 별도 프로젝트가 아니라 **같은 레포의 새 표면** (권고)

이 유스케이스의 가치가 정확히 우리 moat(바이트 보존·게이트)에서 나온다 — 분리하면 그 보증을 잃는다.
형태: ① `auto-hwp fill --template 양식.hwpx --data 명단.csv --map fill-map.json --out zip` CLI
서브커맨드(v1) ② fill-map은 인스펙션 커맨드가 초안 생성 + 사람이 검수(AI 보조는 선택) ③ 스킬/MCP에
동사 추가 ④ (후속) hwp-lab에 "벌크 채움" 탭. 전부 로컬 — 서버 0 원칙 유지.

## 입력 포맷·검수 UX 설계 (2026-07-23 추가 리서치 — 코퍼스 실측 + 생태계 조사)

### 코퍼스 실측 (실물 49종 fill-map 자동 유도 스윕)
- 라벨→값칸 유도 가능 **30/49(61%)** — 유도 0의 19종은 보도자료/공고문류(채움 대상 아님). **양식·
  신청서류는 상위 전부 커버**(혁신바우처 25쌍 · 수행계획서 20쌍 · 투자형 18쌍…).
- 값칸 위치의 **96%가 라벨 우측**(143/149), **예시 텍스트 칸(80) > 빈칸(63)** — "빈칸 전제" 금지 재확인.
- **중복 라벨 37건** → 라벨 매칭만으로는 모호. 결론: "inspect가 라벨 기반 초안 유도 → 검수에서 명시
  주소로 잠금(pin) → fill은 결정론 적용"의 2단 구조 필수.

### fill-map v1 스키마 (hwpx-mcp-server `hwpx.formfill.v1` 구도 채택 + 국내화)
- `schema: "autohwp.fillmap.v1"` · `fields[] = { key, target:{kind,…}, pin, required }`.
- `target.kind`: `label-right`(기본·무수정 원본) · `cell` · `replace`(본문 치환) · `placeholder`
  (`{{key}}` — 우리가 만든 템플릿용) · (후속) `form-field`(누름틀 op 이후).
- **`pin` = 검수에서 잠근 명시 주소**(중복 라벨 해법): inspect가 라벨로 초안 → 검수 확정 → 벌크
  실행은 pin만 신뢰(라벨 재탐색 없음 — 결정론).
- **키는 한글 그대로**(airmang은 placeholder 키 ASCII 전용 — 따라가지 않음. pyhwpx·모두싸인 전부
  한글 헤더 관례). 명단 = "1행=1문서, CSV/XLSX 헤더=키"(업계 단일 컨벤션) + `filename_pattern`.
- 누락·오버플로 **3단 정책**(조사 결론 — "조용한 빈칸은 아무도 안 씀"): 기본=생성+`needsReview[]`
  (사유코드) · `--strict`=행 스킵 · 조용한 빈칸/잘림 금지. **차별점: 실조판으로 zip 전 행별
  오버플로 판정**(쪽수 기준선 대조 — 텍스트 폭 근사 추정 대비 정확).

### 검수 UI 설계 (조사 패턴 3층 + 캐러셀)
- ① 행 그리드(문제 행 배지 우선·필터) ② 본문 좌=채운 페이지 실렌더/우=채운 값 패널(키·값·주소·이전
  예시값), ◀▶ 인원 넘기기(=캐러셀) ③ 승인은 "N건 확인·zip" **배치 1회**(행별 승인은 어떤 제품도
  안 함) + 행별 건너뛰기/값 수정. **채운 셀 인-도큐먼트 하이라이트는 조사상 어느 제품에도 없음**
  (Word 프리뷰조차) — 지오메트리 소유로 우리만 가능한 최대 체감 차별점. 목업:
  `assets/brand/bulk-review-mock.html`(실렌더 인라인 — 사용자 전달됨).

## v1 구현 (2026-07-23 — CLI 2 서브커맨드, 커밋됨)
`crates/auto-hwp-cli/src/fill.rs` — 편집 레인은 웹과 동일한 `hwp_mcp::apply_intent_json`(규칙 한 벌).
- **`auto-hwp inspect 양식 --out fill-map.json`**: 라벨 렉시콘 스캔 → v1 초안(pin+example+ambiguous
  표시). 실물 신청서 실측: 7필드 유도·중복 6종 정확 표시.
- **`auto-hwp fill 양식 --map … --data 명단.(json|csv) --out 디렉토리 [--strict] [--pattern]`**:
  인원별 SetTableCell(pin)/Replace → HWPX + **재개봉 검증**(값 존재 + 쪽수 == 무편집 왕복 기준선) →
  개별 파일 + output.zip + report.json(행별 사유코드). 실물 E2E: 5명 중 필수 누락 1명 —
  기본 모드 5부 생성+needsReview 1, strict 4부+스킵 1. CSV는 단순 파서(따옴표 발견 시 정직 거부 —
  JSON 권장). 유닛 3(파일명 보간/CSV 거부/열수 불일치) · clippy · 게이트 8==8.

## 필드 스튜디오 (2026-07-23 — 사용자 방향: "영역 지정→네이밍→규정→규격화" UX)
/bulk 2단계를 문서-비주얼 스튜디오로 재설계(구현·e2e 잠금 완료):
- **영역 지정**: 실렌더 페이지(◀▶ 전 쪽) 위 셀 클릭 → 결정론 매핑(colBoundaries/rowBoundaries
  히트테스트, LLM 0콜). 자동 인스펙션 초안은 오버레이 박스+이름표로 표시, 클릭=선택(초록)/새 셀=추가
  (좌측 인접 라벨을 이름 초안으로). 병합 커버 셀·표 밖은 정직 안내.
- **네이밍**: 필드 카드에서 인라인 이름 편집 — 이름=명단 헤더/키 매칭 계약(한글 그대로).
- **규정(spec)**: 타입(텍스트/날짜/전화번호/사업자번호/숫자·금액 — 정규식) + 필수 토글. 위반은 채우되
  `format_mismatch`/`missing_required`로 검수에 보고(조용한 통과 금지).
- **규격화**: "규격 저장" = `양식명.fillmap.json`(autohwp.fillmap.v1 + additive `spec` + 템플릿
  sha256 지문) 다운로드 / 불러오기(지문 불일치 시 경고) — **한 번 규정한 양식은 다음 배치에 재사용**.
- **명단 4형식 자동 감지**: CSV · TSV(엑셀 붙여넣기) · "키: 값" 블록 txt(손으로 쓰기 최선) · JSON
  + 파일 열기(.csv/.txt/.json — UTF-8 실패 시 EUC-KR 폴백: 한국 엑셀 CSV 실무 대응).
- 실드라이브 검증: 실물 신청서 7필드+클릭 추가 1·규격 저장·kv 명단 3명·전화번호 형식 위반/필수 누락
  정확 보고·zip — 에러 0. e2e `bulk-fill-073.spec.ts` 잠금.
- **LLM/BYOK는 선택 보조로 설계**(다음 배치): ① 이름·규정 초안 제안(표 텍스트만 전송, BYOK 직접 호출
  — 정적 데모에서도 서버 0 유지) ② 자유 텍스트 명단→구조화 파싱. 핵심 경로는 계속 결정론.
