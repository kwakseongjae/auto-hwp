## 판정

1. **증분 조판(XL) 착수 불필요 — 보류.** 편집→화면이 ~1ms/쪽 선형: 실물 공문서 구간(≤41p)에서
   16~20ms, 130p에서도 136ms — 워커 스레드(FG-14)라 UI 비차단이고 챗/수동 편집 UX에 충분.
   외삽 시 ~300p에서 ~300ms — **재평가 임계 = 150p+ 실물이 실사용에 등장할 때.**
2. **첫 병목은 조판이 아니라 undo 스냅샷 메모리.** 편집 50회(LIVE_UNDO_LIMIT) RSS 증가가
   130p에서 **403MB**(스냅샷당 ~8MB 딥카피) — 수백 페이지·장수 세션에서 브라우저 탭 OOM 위험.
   반면 같은 문서의 **직렬화 크기는 0.2MB**(toHwpx 27.8ms): **스냅샷을 딥카피 대신 HWPX 바이트로
   바꾸면 ~40배 절감**(복원 = 재파싱 33ms — undo 는 드문 조작이라 수용 가능. 자동저장이 이미 같은
   직렬화를 돌고 있어 비용 중복도 아님). → **후속 이슈 후보(엔진 hwp-ops/hwp-mcp 스냅샷 레인).**
3. open/첫 조판/전페이지 렌더는 130p에서도 합계 ~110ms — 문제 없음.

## 정직 단서
- Node wasm ≈ 브라우저(동일 바이너리·V8)지만 워커 RPC 왕복·DOM 주입 비용은 미포함(각 ~수 ms).
- 합성 문서는 표-우세 구성의 **합성**이다 — 100p+ 실물 미확보(068 수집에서 최대 41p). 실물 등장 시 재측정.
- rss 수치는 GC 잡음 포함 근사. 합성 86p→재개봉 70p 편차 = .hwp 유래 from-scratch 직렬화의
  알려진 재조판 축(정확도 이슈 아님 — 페이지네이션 리플로).
- 재실행: `node packages/engine/bench/large-doc-bench.mjs [파일들] [--synth N 기반파일]`
  (합성본은 `corpus/private/bench-synth/` — git 제외).
