# benchmark1 표 행별 높이 실측 (issue 020 — 1페이지 스필 19→18)

우리 자체 레이아웃 엔진이 benchmark1.hwp 를 **19쪽**으로 흘려보내던 원인(한컴 원본은 18쪽)을
행 단위로 실측해 특정한 기록. 결론부터: **두 개의 측정된 기전**을 고쳐 19→18 (한컴과 일치)로
맞췄고, benchmark.hwp 게이트(8==8, 98.9%)는 불변.

- 기전 A (채택): 셀 문단 **마지막 줄의 줄간격 여분(leading)** 을 우리가 과다 예약 → 제거.
- 기전 B (측정 후 발견): 한컴의 **고정/최소 행 높이**(stored cell height)를 우리가 미반영 →
  기전 A 수정이 이를 드러내 17로 언더슈트하므로, stored 높이를 행 높이 **바닥(floor)** 으로 존중.

두 기전 모두 **전역 스케일도 매직넘버도 아니다**(행별 실측값·문서 저장값 기반). place.rs 와
NaiveLayout 은 공유 헬퍼로 **동일하게** 바뀐다(LOCKSTEP).

## 1. 진단 도구 (tracked, 상비)

```
cargo run -q -p auto-hwp-cli --features "shaper rhwp" -- layout-check <file> --rows <section>/<block>
```

지정 표(우리 lift SemanticDoc 의 section/block 인덱스; 오라클이 도는 그 블록 순번)에 대해 **행별
대조표**를 출력한다. 1×1 프레임 래퍼(자가진단표)는 페이지네이션과 동일하게 내부 표로 unwrap 해서
실제 페이지 수를 좌우하는 행을 본다.

컬럼:
- `우리예약` = `table_row_heights[r]` (place.rs::row_heights 와 동일한 값, floor 반영).
- `줄 · ΣEM · 간격 · Σ간격줄 · 패딩` = 그 행을 결정한(가장 큰) 셀의 분해:
  줄 수, Σ bare-EM(vert_size), 줄간격 비율, 실제 예약 줄advance(마지막 줄 leading 트림 반영), CELL_PAD.
- `한컴셀h` = rhwp 가 파싱한 stored cell height / row_span (한컴의 실제 행 기여분).
- `한컴내용` = 그 행 최대 콘텐츠 = lineseg 수직 span(`last.vertical_pos+line_height − first.vertical_pos`,
  vertical_pos 미기입 시 Σ line_height 폴백).
- `한컴슬랙` = `한컴셀h − 한컴내용` (실 패딩 + 고정행 여유).
- `선언패딩` = 결정 셀의 **선언된** 세로 패딩(top+bottom, apply_inner_margin 반영) — 가설(c) 직접 검증.
- `한컴줄seg / seg` = 그 셀의 Σ line_height (leading 제외 bare boxes) / lineseg 개수.
- `델타` = `우리예약 − 한컴셀h` (>0 = 우리가 과다 예약).

블록 스캔 결과(section 0 의 표들): 프레임 1×1 이 다수. 사용자 QA 가 지목한 "block 6, 17행
체크리스트"는 **우리 모델에서 block 11 (전역 표순번 4) = 자가진단표 17행×3열**(1×1 프레임 안의
중첩 그리드)에 해당한다. block 6 은 1행×4열의 다른 표였다(이슈-코드 어긋남 → 코드 기준으로 진행).

## 2. 실측 — 스필 표(block 11 자가진단표) + 대조군

### 2.1 수정 전 (19쪽) — block 11, 행별 (발췌)

```
행 |  우리예약   줄  ΣEM   간격  Σ간격줄  패딩 | 한컴셀h  한컴내용  선언패딩  seg |    델타
 4 |    1490    1  1100  1.10   1210   280 |  1382    1100     282    1  |   +108
 5 |    2700    2  2200  1.10   2420   280 |  2594    2200     282    2  |   +106
 8 |    1490    1  1100  1.10   1210   280 |  1382    1100     282    1  |   +108
합계:  우리 59060 · 한컴(셀h) 52144 · 총델타 +6916 (행당 평균 +407)
```

체계적 패턴 (줄 수 일치 행): **우리는 모든 줄에 줄간격 1.10 을 곱해 예약**하는데(1100×1.10=1210),
**한컴 내용은 bare EM 1100** 이다. 즉 한컴은 마지막(유일) 줄에 leading 을 얹지 않는다.
그리고 **선언패딩 282 ≈ 우리 CELL_PAD 280** — 패딩은 맞다.

2줄 행(row 5)으로 검증: 한컴셀h 2594 = ΣEM 2200 + 1×(1100×0.10=110) leading + 282 pad ≈ 2592.
즉 **한컴 = ΣEM + (줄수−1)×EM×(ratio−1) + 패딩** (줄 사이 gap 에만 leading, 마지막 줄엔 없음).
우리 = ΣEM×ratio + CELL_PAD = 한컴 + **EM×(ratio−1)** (마지막 줄 leading 1개분 과다).

### 2.3 가설 평가

- **(a) linespace 적용 방식** — 부분 관련. 한컴 lineseg line_height = bare EM 이고 줄간격은
  줄 **사이** advance 에만 들어간다. 우리는 마지막 줄까지 곱했다. → (b)로 수렴.
- **(b) 마지막 줄 leading 미가산** — **채택**. 행당 상수 과대(≈ EM×(ratio−1))가 정확히 이 패턴.
  block 11 단일줄 행 델타 +108 = 1100×(1.10−1)+ (280−282). 2줄 행도 ±2 로 일치.
- **(c) CELL_PAD(세로 280) 불일치** — **기각**. 선언패딩 실측 = 280~282 로 우리 값과 일치.
  block 18 의 큰 슬랙(790~2445)은 패딩이 아니라 **고정 행 높이**(han_cell_h 2990 이 내용
  1100~2200 과 무관하게 일정)였다 → 새 기전 B.
- **(d) ascent/descent(폰트 메트릭) 차이** — **기각**. 델타가 폰트 크기 비례가 아니라 줄 수에
  선형(줄당 상수)이며 단일줄에서 EM×(ratio−1) 로 정확히 설명됨.

### 2.4 새 기전 B — 고정/최소 행 높이 (block 18)

block 18 은 `선언패딩 = 280`(우리와 동일)인데 `한컴셀h 2990` 이 내용(2000/2200/1100)과 무관하게
고정이다. 이는 **한컴이 저장한 행 높이(stored cell height)** — 사용자가 지정한 고정/최소 행 높이다.
우리는 내용 기반으로만 예약해 이 행들을 **과소** 예약한다. 수정 전엔 (a)/(b)의 마지막-줄 leading
과대가 이 과소를 우연히 상쇄했다. 기전 A만 고치면 상쇄가 사라져 benchmark1 이 **17로 언더슈트**한다
(실측: A 단독 = 17). 따라서 stored 높이를 행 높이 floor 로 존중해야 18 이 된다.

## 3. 국소 수정 (place.rs ↔ NaiveLayout 동일)

### 기전 A — 셀 문단 마지막 줄 leading 트림
`hwp_typeset::cell_paragraph_height` (신규 공유 헬퍼) — 셀 문단 높이 =
`위/아래 간격 + Σ(vert_size×ratio) − (마지막 줄 vert_size × (ratio−1))`.
`lib.rs::block_height` 와 `place.rs::block_height_for_place` **둘 다** 이 헬퍼에 위임 → LOCKSTEP.
마지막 줄 leading 은 그 줄 **잉크 아래의 빈 공간**이라 예약에서 빼도 글자를 자르지 않는다(잉크
extent = content_new). 본문 페이지네이션(NaiveLayout 의 문단 스택)은 이 경로를 쓰지 않으므로
**표 셀에만** 국한된다.

### 기전 B — stored cell height 를 행 높이 floor 로
`lift.rs::stored_row_heights` — rhwp cell.height/row_span 을 `Table::row_heights` 로 리프트.
기존 `apply_row_overrides` 가 이를 **floor(max(content, stored))** 로 존중한다(place.rs 와
lib.rs 양쪽에서 호출 → LOCKSTEP). 내용 ≥ stored 인 행은 무변화, 고정행만 stored 로 올라간다.

두 기전은 **상보적·둘 다 필요**:
- floor 단독 = 19 (block 11 이 여전히 과다, floor 는 낮추지 못함).
- 트림 단독 = 17 (block 18 고정행이 드러나 언더슈트).
- 트림 + floor = **18** (한컴 일치).

# 자가진단표 프레임 구조 실측 (issue 024 — 상자 3종의 실체 + 귀하 이탈)

020 은 자가진단표(**block 11**)를 **행 높이** 축으로 실측했다. 024 는 같은 프레임을 **상자(테두리)와
콘텐츠 이탈** 축으로 실측한다. 결론부터: 아키텍트의 두 가설이 **실측으로 뒤집혔다** —
(A) 상자는 "문단 테두리"가 아니라 **중첩표의 셀 테두리**(이미 충실히 lift됨)다. (B) "귀하"는
드롭되지 **않는다**(18쪽 어디에도 없다던 grep은 own-render 가 글자당 `<text>` 하나를 쓰는 탓의
착시). "귀하"는 렌더되지만 **외곽 상자를 벗어난다**. 진짜 기전은 **셀 그리기(place_cell_content)가
예약(cell_paragraph_height)보다 문단 후행 leading 만큼 더 내려 긋는 것**이다.

## 7. 구조 실측 — 프레임 내부 블록 시퀀스

`block 11` = **1×1 프레임 래퍼** (stored cell height 50828) → 그 셀 안에:
- `[0]` 빈 앵커 문단 (pshape 102)
- `[1]` **17×3 중첩 그리드** (자가진단표 본체) — 페이지네이션은 이 그리드를 promote(`unwrap_frame_table`)

| r0 | "※ 본 내용은 사업 신청자격 유무를…" + 앵커 + **1×2 중첩표** | 3 blocks |

"귀하"는 별도 블록이 아니라 **r16 큰 셀의 11번째(마지막) 문단**이다. QA (1)(동의행 상자),

## 8. 상자 3종의 실체 — 셀 테두리 (문단 테두리 아님)

### 8.1 원인 A 반증: `ParaShape.border_fill_id` 는 문단 테두리가 아니다
rhwp 파싱 실측: benchmark1 의 **176개 para_shape 전부**가 `border_fill_id ∈ {1,2,3}` 을 가지며
`border_spacing = [0,0,0,0]`. 그 border_fill 1/2/3 의 4변은 **전부 `None`**(선없음)이다 — 즉
문서 기본 "테두리 없음" fill 이다. 이걸 문단 테두리로 lift 하면 **거의 모든 문단에 상자**가 그려진다
(파국). ⇒ **`ParaShape.border_fill_id` 소비는 이 문서의 상자와 무관**하며, 소비해도 그릴 게 없다.
(그래서 024 는 lift_para_shape 를 건드리지 않는다 — 이슈의 "원인 A 경로(문단테두리면 lift+render)"
전제가 거짓으로 실측됨.)

### 8.2 상자는 중첩표 **셀 테두리** — 이미 충실히 lift됨
17×3 중첩표의 rhwp 셀 `border_fill_id → [L,R,T,B]` (우리 lift 결과와 **정확히 일치**):

```
r0  bfid=55 [-,-,-,-]        (상자 없음 — 한컴도 없음)
r1  bfid= 5 [-,-,-,-]        (헤더 [신청자 체크항목] — 한컴도 상자 없음!)
r2  bfid=45 [-,-,-,Solid]    (아래변만)
r3..r13 c0 bfid=36 [Solid,-,Dot,Dot]  c1 [-,Dot,Dot,Dot]  c2 [Dot,Solid,Dot,Dot]  (체크 그리드)
r14 bfid=37 [Solid,-,Dot,Solid] …      (그리드 마지막 — 아래변 Solid)
r15 bfid=25 [-,-,Solid,Solid]
r16 bfid= 6 [Solid,Solid,Solid,Solid]  (동의/귀하 — 4변 완전 상자)
```

즉 **QA(1) "동의…신청인" 행(r16)은 4변 상자를 이미 가진다**(그리고 그려진다). **QA(2) "[신청자
체크항목]" 헤더(r1)는 한컴 원본에서도 셀 테두리가 없다**(bfid=5, 4변 선없음) — 우리 렌더가 원본과
일치. 원본 rhwp 네이티브 렌더(ground truth) 스크린샷에서도 헤더는 상자가 아니라 그냥 텍스트다.
그러므로 "상자 누락" 버그는 없다(013/R1 의 per-edge 셀 테두리 lift 가 이미 충실). **남은 실체는
QA(3): r16 상자가 콘텐츠를 다 감싸지 못해 귀하가 삐져나오는 것**이다.

## 10. 수정 (place_cell_content 후행 leading 트림) + 검증
