## 0. 요약 — 한 문단 판정

렌더/조판/round-trip **코어의 충실도는 벤치마크 범위 안에서 강하다**(8==8·18==18, 줄바꿈 98.9%).
지금 단계의 병목은 코어가 아니라 **양쪽 끝**에 있다: (i) **최종 사용자 쪽** — AI가 문서를 "이해"하는
층이 구조적으로 없어서(문서 프로필·요약·의도 추론 0) 사용자가 앵커를 찍고 문서 맥락을 매번 설명해야
하고, AI가 낼 수 있는 편집이 엔진 41종 중 15종뿐이며, BYOK 없이는 AI가 아예 꺼져 있다.
(ii) **임베드 개발자 쪽** — npm 미발행 + 수동 배선 6단계(wasm 4파일 복사·CSS·프록시 자가구축·폰트
주입·CSP·SSR off)로 "설치 5분" 경험이 아직 없다. (iii) **전략 축** — rhwp upstream이 4버전 앞서며
전체 스택 정면 경쟁자가 됐고, 정부 HWPX 의무화(2026-05-18)로 HWPX 입력의 전략 가치가 급등했다.
엔진 고유의 구조적 상한(폰트 치환, 근사 조판, 편집마다 전체 재조판, 수식/차트 PDF 스텁, .hwp
재저장 불가, 실 한컴 미검증)은 실사용 문서가 복잡해질수록 드러난다.

## 1. 구성과 동작 원리 — 어디가 사용자 경험의 어디에 닿는가

```
[업로드] ── hwp-ingest(포맷감지·리소스한도)
   ├─ .hwp  → external/rhwp v0.7.15(파스전용) → hwp-rhwp lift → SemanticDoc(IR)
   └─ .hwpx → hwp-hwpx 자체 파서(subset 모델 + 나머지 verbatim 보존 = round-trip moat)
[조판] hwp-typeset place_doc — 기본 ApproxFontMetrics(근사), shaper feature시 rustybuzz 실측
[렌더] hwp-render PaintOp 4종(Glyph/Rect/Line/Image) → SVG    [편집] hwp-ops Op-bus(41 Intent)
[export] hwp-export — HTML(semantic-reflow)·PDF(krilla)·HWPX(dirty-only 재emit)
[셸] hwp-session(px 지오메트리) → hwp-wasm(9.3~9.8MB) → editor-core(headless) → react UI
[AI] ChatPanel → buildDocContext(마킹 앵커+표 그리드만) → /api/hwp-edit(BYOK 프록시) → Intent[]
```

사용자 경험과의 대응: **업로드 성공률**=ingest+rhwp/hwpx 파서 커버리지 → **"원본처럼 보이나"**=
조판 메트릭+폰트 번들 → **"만질 수 있나"**=lift가 IR로 올린 것만(Raw 인라인은 뷰 전용) →
**"내보내면 온전한가"**=export별 커버리지 차이 → **"말로 시키면 되나"**=docContext+화이트리스트.

## 3. 대표 시나리오 워크스루 — "사업계획서를 업로드했다"

1. **업로드**: .hwp/.hwpx면 대부분 열림(065 이후 실물 24/24 급 회복). 암호 걸린 파일·아주 옛
   파일은 정직 거부. ✅ 강함 / ⚠️ U10 잔존.
2. **첫 화면**: ~12MB 로드 후 렌더. 서체가 함초롬이면 나눔으로 치환돼 "비슷하지만 다른" 인상(U8).
   차트/도형은 보임(뷰 전용). 대체로 ✅.
3. **"이 표 채워줘"라고 치면**: **여기서 막힌다.** 표를 클릭해 마킹하지 않았으면 좌표가 없어
   실패(U2). 마킹해도 모델은 이 문서가 사업계획서인지, 어떤 항목을 요구하는 양식인지 모른다(U1)
   → 사용자가 배경을 구구절절 쓰거나, 여러 턴 왕복. 6턴 넘으면 앞 설명 유실(U5).
4. **"전체적으로 볼드 좀 빼줘 / 여백 줄여줘"**: 화이트리스트 밖이라 불가(U4). 수동으로.
5. **PDF로 제출**: 표·텍스트는 온전, 문서에 차트가 있었다면 스텁 박스(U7). 심사처에 그대로 제출
   했다가 발견하는 최악 경로가 열려 있음 — **export 시 경고조차 없음**.
