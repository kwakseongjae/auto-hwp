# auto-hwp — 프로덕션 진단 & 로드맵 (2026-06-18)

목표: **한글(Hancom Office) 데스크탑앱을 대체**하는 프로덕션 앱 + **AI가 모든 걸 하는** 서비스.
이 문서는 현재 구현 상태에서 거기까지의 거리를 정직하게 진단한다.

---

## 1. 현재 상태 (정직하게)

### 강점 — 진짜 해자(moat)가 있는 부분
- **변환기 (.hwp → .hwpx)**: 텍스트 · 글자서식(굵게/기울임/크기/색/밑줄/취소선/첨자) · 스크립트별 폰트 · 표(중첩·병합·다문단·음영) · 페이지(크기·여백·방향) · 다중 구역 · 이미지 · 수식 · 하이퍼링크 · 각주/미주 · 머리말/꼬리말 — oracle(LibreOffice+H2Orestart) 검증.
- **라운드트립 바이트-안정성 moat**: 원본 HWPX를 verbatim passthrough + dirty-only 재방출. 생태계 #1 버그(라운드트립 데이터 손실)를 구조적으로 해결.
- **자체 소유**: 포맷-중립 SemanticDoc AST + 자체 HWPX 직렬화기 + typed op-bus(InsertText/DeleteRange/SetCharPr/InsertImage/Table*/MergeCells/Append*/SetPageLayout) + EditSession(undo/redo) + open-safety 검증기.
- **AI seam**: LlmProvider(BYOK) + propose→commit + 구조보존 doc→Markdown(anti-PDF-flatten RAG) + 임베드 MCP 제어서버.

### 약점 — 한글 대체까지 먼 부분
- **자체 레이아웃 엔진 없음**: `hwp-typeset`/`hwp-render`는 36줄 스텁. 렌더링은 **100% rhwp 부트스트랩 의존**.
- **자체 출력을 인앱으로 못 그림**: 우리 변환 .hwpx는 LibreOffice에선 깔끔하지만 rhwp(인앱 렌더러)는 linesegarray가 없어 overflow. → **자기 출력조차 인앱 미리보기 불가**.
- **WYSIWYG 편집기 아님**: op 표면은 풍부하나 **편집 캔버스(캐럿/선택/IME)가 없음**. 뷰어는 rhwp SVG 페이지 뷰 + 커맨드 스트립 + AI이지, 워드프로세서가 아님.
- **실제 한컴 한글에서 한 번도 안 열어봄**: oracle는 LibreOffice 프록시. **최대 미검증 리스크**.

---

## 2. 크리티컬 패스 (가장 중요한 진단)

**#1 관문 = 자체 레이아웃/셰이퍼 엔진 소유** (`hwp-typeset`, 현재 스텁). 이것이 모든 하류를 막는다:

```
레이아웃 엔진 (줄바꿈+금칙+페이지네이션+폰트 셰이핑 → linesegarray 생성)
        │
        ├─→ 자체 출력의 충실 인앱 렌더 (rhwp linesegarray 의존 탈피)
        ├─→ WYSIWYG 편집 캔버스 (캐럿 위치 = 화면↔문서 좌표 = 레이아웃 필요)
        └─→ rhwp 부트스트랩 독립 (장기 라이선스/통제 리스크 해소)
                │
                └─→ 자유 편집 → AI-everything (실제 편집기 위의 에이전트)
```

레이아웃 없이는 진짜 편집기(캐럿이 글리프 위치를 알아야 함)도, 충실 인앱 미리보기도 불가능. **레이아웃이 중심 프런티어 — 다개월(multi-month) 작업.**

## 4. AI North Star (단계화)

현재 propose→commit + MCP가 토대. "AI가 모든 걸"은 **점진적으로** 가능:
1. (완료) 제안→미리보기→커밋, BYOK, 구조보존 RAG.
2. **풍부한 AI op 표면** — op-bus는 이미 풍부(InsertText/Delete/Table*/…); 임의 위치 삽입·재구조화·서식까지 노출.
3. **에이전트 루프** — MCP 제어면 위에서 plan→act→verify(문서 위 자율 편집).
4. **RAG/멀티문서** — 문서 구조 + 여러 문서에 걸친 검색·답변.
5. **대화/음성 주 인터페이스**.

의존성: P2(편집 표면)이 "파워유저가 하는 모든 것"의 전제. 단, **AI-authoring은 지금 seam 위에서 이미 출시 가능**(별도 가치).

## 5. 지금 당장 (immediate next, 레버리지 순)

1. **실제 한컴 한글 수용 테스트** — 가장 큰 미검증 리스크 해소. 변환 .hwpx를 실제 한글(Windows)에서 열기. 싸고 레버리지 최고 (moat 전체를 de-risk).
2. **레이아웃 엔진 스파이크** (P1 착수) — 한 문단의 줄바꿈+페이지네이션 → linesegarray 생성 → rhwp가 우리 .hwpx를 제대로 렌더하는지 폐루프 확인. 다개월 P1을 de-risk.
3. **대규모 코퍼스 변환 감사** — 수백 개 실 .hwp 변환, clean/degrade/crash 측정, 빈도순 fidelity 수정 우선순위.

## 6. 최대 리스크
1. **실 한컴 미검증** — moat가 실제 타겟에 대해 미검증.
2. **레이아웃 엔진이 다개월** — 크리티컬 패스가 가장 어렵고 긴 조각. 과소평가 위험.
3. **rhwp 포크 의존** — 라이선스/유지보수 부채 + 인앱 렌더러가 자체 출력을 못 그림. 장기적으로 레이아웃 소유로만 해소.
4. **폰트 라이선스** — HY/한컴 독점 폰트; FOSS 폴백 → 한글과 렌더 차이.
5. **신뢰/데이터 손실** — 워드프로세서가 데이터를 잃으면 끝. 라운드트립 moat가 돕지만 코퍼스-스케일 + 크래시-안전 증명 필요.
6. **자유 WYSIWYG 편집은 거대한 빌드** — "변환+패치"에서 "워드처럼 편집"까지가 워드프로세서의 본체.

## 7. 정직한 평가 — 한글 대체까지의 거리

**변환기 + AI-authoring**은 강력하고 차별화됨 — "AI-네이티브 HWPX" 각도에서 진짜 앞서 있음. **지금 제품화 가능한 가치**: ".hwp를 편집가능한 .hwpx로 변환 + AI로 작성/편집" 도구.

하지만 **한글을 대체하는 WYSIWYG 데스크탑 편집기**로는 아직 초기다: 자체 레이아웃 엔진 없음(중심 프런티어, 다개월), 편집 캔버스 없음, 렌더는 rhwp 의존(자체 출력조차 못 그림), 실 한글 미검증.

**스마트한 전략**: 변환+AI 가치를 **지금 출시**하면서, 레이아웃→캔버스→편집을 병렬로 쌓아 전체 대체로 간다. 크리티컬 패스는 명확하다 — **레이아웃 엔진**.
